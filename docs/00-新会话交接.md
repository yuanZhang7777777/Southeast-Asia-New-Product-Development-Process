> 日期：2026-06-29

## 1. 项目一句话定位

本项目要做的是“东南亚海外仓新品流程协同平台”。

它不是替代 ERP、销售系统、运营系统，而是在现有系统和钉钉在线表之上做一个流程协同层，把当前分散在在线表里的分配、认领、审核、备货、到货、刊登、提醒、监控和四周复盘变成可追踪的系统流程。

## 2. 当前共识

1. 第一版不要一次性替换所有钉钉在线表操作习惯。
2. 第一版应尽量贴近原来的表格逻辑，先解决责任人、状态、截止时间、提醒和进度追踪。
3. 酷应用作为钉钉内的处理入口。
4. 钉钉待办用于必须处理的任务，例如销售认领、主管审核、备货确认、四周总结。
5. 钉钉工作通知用于提醒、异常、汇总和升级。
6. ERP、销售系统、运营系统已有的数据尽量同步过来，不让人重复填。
7. 人工只填写需要业务判断的字段，例如是否认领、认领单销、不认领原因、审核意见、调研结论、复盘总结。

## 5. 当前主流程

```text
待认领数据
  → 分配认领任务
  → 销售认领
  → 主管审核
  → 开发备货
  → 销售备货
  → 备货监控
  → 到货监控
  → 二次调研
  → 具体调整
  → 刊登运营
  → 1-4周监控实际达成情况
  → 四周总结
```

其中已经识别出的主要分支：

- 销售认领：认领填单销 / 不认领填理由
- 主管审核：审核通过 / 审核驳回
- 到货监控：集团内部认领监控 / 其他集团到货监控
- 二次调研：外部老品热销监控 / 老品销量下跌 / 销售自主调研
- 刊登运营：缺货监控 / 广告费监控
- 四周总结：实际达成时缺货监控

## 6. 和技术人员沟通时的推荐说法

可以这样描述：

> 我们这次不是单纯把在线表换成网页表单，而是做一个新品流程协同层。现在 ERP、销售系统、运营系统分别有各自的数据和操作，但新品流程推进主要还散在钉钉在线表里，导致责任人、状态、提醒和复盘不好追踪。  
> 第一版希望先把待认领、分配、销售认领、主管审核、备货、到货、刊登运营、四周总结这些关键节点串起来。已有系统能提供的数据尽量同步，人工只补判断类字段。钉钉酷应用作为入口，必须处理的事情走待办，异常和汇总走工作通知。

## 7. 下一轮重点要讨论的问题

优先讨论这些，不要一开始陷入所有字段细节：

1. 这个新品流程平台是独立做，还是作为现有运营系统的一个模块。
2. ERP、销售系统、运营系统分别能提供哪些接口或数据表。
3. 待认领数据从哪里来，是否能自动入池。
4. 分配任务按什么规则分配。
5. 销售认领必须填哪些字段。
6. 主管审核是否必走。
7. 备货申请由谁填，谁最终确认数量。
8. 到货后由谁处理刊登、库存、售价调整。
9. 第一版钉钉提醒只做哪些节点。
10. 平台提交后是否要写回钉钉在线表。

## 8. MVP 建议范围

第一版先做最小闭环：

```text
待认领数据
  → 分配认领任务
  → 销售认领
  → 主管审核
  → 备货确认
  → 到货提醒
  → 四周总结
```

第一版先不做太复杂的广告费、缺货、1-4 周精细化监控，这些可以放第二阶段。
