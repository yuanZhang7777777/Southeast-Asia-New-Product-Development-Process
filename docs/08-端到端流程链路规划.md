# 东南亚新品流程端到端链路规划

> 日期：2026-06-29  
> 状态：端到端链路规划草案，供业务和技术共同对齐  
> 关键修正：第一阶段业务验收只做到 `入池 -> 分配 -> 销售认领 -> 主管审核`，后续链路先规划清楚，不假装已经闭环。

## 1. 总体原则

这套系统要覆盖完整新品生命周期，但不能第一版就把所有链路做成半截功能。

正确拆法：

```text
先把前置决策闭环做扎实：
入池 -> 分配 -> 销售认领 -> 主管审核

再逐段接后续执行链路：
备货确认 -> 采购/下单 -> 备货监控 -> 到货/入仓 -> 刊登运营 -> 1-4 周监控 -> 四周总结 -> 后续动作
```

第一阶段的目标不是“自动完成备货到复盘”，而是解决最前面最容易混乱的四件事：

- 哪些数据进入新品机会池。
- 谁负责分配。
- 销售怎么认领或拒绝。
- 主管怎么审核并形成明确结论。

## 2. 完整链路总览

```text
01 数据入池
  -> 02 机会去重与分组
  -> 03 分配任务
  -> 04 销售认领
  -> 05 主管审核
  -> 06 开发/供应链确认
  -> 07 销售/组长备货确认
  -> 08 采购/下单
  -> 09 备货执行监控
  -> 10 发货/在途监控
  -> 11 到货/入仓/上架
  -> 12 刊登/库存/售价确认
  -> 13 上线后 1-4 周监控
  -> 14 异常调整/二次调研
  -> 15 四周总结
  -> 16 后续动作：补货、调价、停推、复议、沉淀经验
```

## 3. 阶段拆分

| 阶段 | 名称 | 目标 | 业务验收边界 |
|---|---|---|---|
| Phase 1 | 前置决策闭环 | 把新品机会从源表变成明确的销售认领和主管审核结论 | 审核通过 / 驳回 / 退回补充 |
| Phase 2 | 备货决策闭环 | 把审核通过的机会转成可执行备货需求 | 最终备货数量、国家、仓库、责任人确认 |
| Phase 3 | 采购与备货执行 | 跟踪采购下单、发货、延期、数量不足 | 采购单 / 备货单状态可追踪 |
| Phase 4 | 到货与刊登运营 | 到货后推动库存、刊登、售价、运营动作 | 到货后有人处理且可追责 |
| Phase 5 | 上线监控与复盘 | 监控 1-4 周实际达成和异常原因 | 四周总结和后续动作闭环 |

## 4. Phase 1：前置决策闭环

### 4.1 数据入池

目标：把可讨论的新品机会统一进入池子。

来源可能包括：

- PH/TH/VN 分站点新品表。
- 财根团队新品开发表。
- 开发认领反馈表。
- 销售选品调研表。
- 直发热销转海外仓确认表。
- 后续 ERP / 销售 / 库存系统识别出的机会。

第一阶段只需要确认：

| 事项 | 建议口径 |
|---|---|
| 入池粒度 | 以子 SKU 为记录，主 SKU 用于整组分配 |
| 来源追踪 | 必须保存源文件、sheet、行号、列段快照 |
| 去重口径 | 先按 `来源 + sheet + 行号` 防重复；业务去重规则另行确认 |
| 字段范围 | 只抽取分配和认领需要的核心字段，其余保存快照 |

### 4.2 机会去重与分组

目标：避免同一个主 SKU 被拆散，或同一个机会重复派发。

需要确认：

- 同一主 SKU 下多个子 SKU 是否必须同一销售负责。
- 同一子 SKU 如果在多个源表出现，哪个来源优先。
- 复议数据是新机会，还是挂到原机会下。
- 其他集团 / 其他平台反馈是否第一阶段只保存快照。

### 4.3 分配任务

目标：主管 / 组长把机会分给销售。

第一阶段建议：

| 规则点 | 第一阶段建议 |
|---|---|
| 分配单位 | 主 SKU 整组分配 |
| 负载统计 | 暂按未完成认领任务数 + 子 SKU 数 |
| 推荐方式 | 系统给建议，人工确认生效 |
| 候选人来源 | 先由组长选择，后续接人员映射 |
| 改派 | 允许改派，但必须留原因和审计记录 |

第一阶段不建议做：

- 按历史业绩自动打分。
- 按类目熟悉度自动派单。
- 多销售抢单。
- 全自动分配。

### 4.4 销售认领

目标：销售给出是否承接和承接依据。

第一阶段必填：

| 场景 | 必填 |
|---|---|
| 认领 | 认领单销、销售反馈说明 |
| 不认领 | 不认领原因、反馈说明 |

待确认：

- 认领单销是日销、周销还是月销。
- 是否必须填竞品链接或截图。
- 不认领原因是否做枚举。
- 销售是否可以改系统带出的售价、成本、备货建议。

### 4.5 主管审核

目标：主管判断销售认领是否可进入后续备货决策。

审核结果建议：

| 结果 | 含义 | 后续 |
|---|---|---|
| 通过 | 认领有效，可进入备货决策 | 进入 Phase 2 |
| 驳回 | 不进入后续链路 | 关闭或归档 |
| 退回补充 | 销售信息不足 | 回到销售认领 |

第一阶段验收结束点：

```text
每条入池机会都有明确审核结论，并能追踪是谁分配、谁认领、谁审核、什么时候处理。
```

## 5. Phase 2：备货决策闭环

Phase 2 不应该只做“备货草稿”四个字，需要拆成明确节点。

```text
审核通过
  -> 系统生成备货建议
  -> 开发/供应链确认成本、供应商、MOQ、交期
  -> 销售/组长确认备货单销、数量、国家、仓库
  -> 主管/PMO 确认最终备货方案
  -> 生成正式备货申请或写入备货申请表
```

必须确认的问题：

- 备货数量谁最终拍板。
- `认领单销 * 30` 是建议值还是默认最终值。
- 成本、体积、供应商从哪里取。
- 是否必须先供应链核价。
- 是否写回 `海外仓备货申请表.xlsx` 或钉钉在线表。

Phase 2 业务验收：

```text
审核通过的机会，可以形成一条责任清晰、数量清晰、国家/仓库清晰的备货申请。
```

## 6. Phase 3：采购与备货执行

```text
备货申请确认
  -> 采购下单
  -> 供应商备货
  -> 发货
  -> 在途
  -> 预计到仓
  -> 异常处理
```

关键角色：

- 开发 / 采购：维护采购单、供应商、交期、发货状态。
- 主管 / PMO：处理延期、数量不足、采购异常。
- 系统：同步 ERP / 采购 / 物流状态。

第一版后置原因：

- 需要明确 ERP 字段和采购单状态来源。
- 需要确认采购/开发当前实际操作表。
- 需要确认异常阈值和责任人。

## 7. Phase 4：到货与刊登运营

```text
到货/入仓
  -> 上架/可售
  -> 提醒销售/运营
  -> 刊登确认
  -> 库存和售价确认
  -> 广告/推品准备
```

必须确认的问题：

- 到货触发点是入仓、上架、可售还是首单。
- 到货数量不足如何处理。
- 到货后谁负责刊登。
- 刊登后 ItemID / 链接从哪里回填。
- 是否需要和其他集团到货监控合并。

## 8. Phase 5：上线监控、异常调整和四周总结

上线后不只是“四周总结”，中间还有三类监控：

```text
第 1 周检查
第 2 周检查
第 3 周检查
第 4 周检查
```

每周需要看：

- 实际销量 / 销售额 / 毛利。
- 库存是否缺货。
- 广告费和 ROI / ACOS。
- 是否按认领单销达成。
- 是否需要补货、调价、停推、二次调研。

四周总结应该沉淀：

- 是否达成。
- 未达成原因。
- 缺货是否影响判断。
- 广告是否影响判断。
- 售价 / 库存 / 刊登 / 竞品变化是否影响。
- 后续动作：补货、继续观察、调价、停止推进、复议。

## 9. 当前系统边界修正

当前技术骨架里已经存在 `stocking`、`arrival`、`summary` 模块，但它们在业务上只能视为占位接口。

第一阶段不要把这些模块作为验收范围：

- 不验收正式备货逻辑。
- 不验收到货提醒逻辑。
- 不验收 1-4 周监控。
- 不验收四周总结。

第一阶段只验收：

- 数据能入池。
- 能按规则分配。
- 销售能认领 / 不认领。
- 主管能审核通过 / 驳回 / 退回补充。
- 全过程有任务状态和审计记录。

## 10. 下一轮会议建议

下一轮不要讨论全链路所有字段，先只确认 Phase 1。

建议议题：

1. 哪些源表先进入机会池。
2. 入池粒度是主 SKU 还是子 SKU。
3. 主 SKU 是否必须整组分配。
4. 分配人是谁。
5. 候选销售从哪里来。
6. 销售认领必填哪些字段。
7. 不认领原因枚举。
8. 主管审核结果有哪些。
9. 审核通过后是否只标记“可进入备货决策”，不立即自动生成正式备货。
10. 是否需要写回原在线表。
