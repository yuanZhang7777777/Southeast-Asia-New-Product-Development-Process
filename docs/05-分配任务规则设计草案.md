# 分配任务规则设计草案

> 日期：2026-06-29  
> 状态：MVP 已实现最小规则，业务口径待确认。

## 1. 当前已实现的最小规则

| 规则点 | 当前实现 | 待业务确认 |
|---|---|---|
| 分组单位 | 按 `main_sku` 聚合预览 | 是否所有主 SKU 必须整组分配 |
| 负载单位 | 用子 SKU 数量衡量任务量 | 是否需要按类目、站点、难度加权 |
| 候选人 | 分配台手工输入候选销售 | 是否从人员映射自动筛选 |
| 推荐逻辑 | 把主 SKU 组分给当前负载最低的候选人 | 是否要加入历史业绩、站点熟悉度、类目熟悉度 |
| 生效方式 | 系统只预览建议，人工确认后生成销售认领任务 | 是否允许系统自动分配 |
| 改派 | 支持按任务改派，并要求填写原因 | 是否需要主管审批或改派次数限制 |

## 2. MVP 分配流程

```text
待分配新品机会
  -> 按 main_sku 聚合
  -> 计算每组子 SKU 数
  -> 输入候选销售
  -> 系统给出均衡建议
  -> 组长人工确认
  -> 为销售生成认领任务
```

## 3. 会议确认清单

| 问题 | 建议默认口径 | 业务结论 |
|---|---|---|
| PH/TH/VN 是否跨站点混合分配 | 不混合，先按站点过滤候选人 | 待确认 |
| 同一主 SKU 是否必须分给同一销售 | 必须整组分配，避免子 SKU 分散 | 待确认 |
| 销售当前负载怎么统计 | 第一版按未完成认领任务数 + 子 SKU 数 | 待确认 |
| 是否考虑历史认领成功率 | 第一版不考虑，只在后台展示 | 待确认 |
| 是否考虑类目熟悉度 | 第一版不考虑，后续做规则配置 | 待确认 |
| 是否允许销售拒绝后自动转派 | 第一版不自动转派，由组长改派 | 待确认 |
| 改派后原任务怎么处理 | 原任务关闭，新任务生成，保留审计 | 待确认 |
| 一条机会能否分给多个销售 | 第一版不允许，多人认领作为后续扩展 | 待确认 |
| 分配后认领 SLA | 建议 24 小时 | 待确认 |
| 逾期升级对象 | 销售本人 -> 直属主管/组长 | 待确认 |

## 4. 后续可扩展但不进 MVP 的规则

- 按站点、类目、历史业绩、当前缺货压力综合打分。
- 黑名单/白名单：某销售不接某类目或优先接某类目。
- 大主 SKU 自动拆组，但需要业务明确拆组边界。
- 批量导入人员容量，如每天最多可接几个主 SKU。
- 主管审批改派。

## 5. 当前代码对应

- 分配预览：`POST /assignments/preview`
- 确认分配：`POST /assignments/confirm`
- 改派任务：`POST /assignments/reassign`
- 核心逻辑：`backend/app/services.py`
- 前端入口：`frontend/src/App.tsx` 的 `分配台`
