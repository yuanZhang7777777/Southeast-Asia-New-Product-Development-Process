# 东南亚新品流程协同平台系统架构重新设计方案

> 日期：2026-06-29  
> 状态：MVP 技术方案已落地为项目骨架，业务细节继续按模块确认。

## 1. 系统定位

本系统是钉钉在线表、ERP、库存、销售数据之上的“新品流程协同层”，不替代 ERP，也不把第一版做成大而全 BI 平台。

第一阶段业务验收只跑通：

```text
入池 -> 分配 -> 销售认领 -> 主管审核
```

`备货草稿`、`到货提醒`、`四周总结` 属于后续链路，当前只保留技术占位和路线图，不作为第一阶段业务验收范围。完整端到端链路见 `docs/08-端到端流程链路规划.md`。

## 2. 技术栈

| 层级 | 选型 | 当前落地 |
|---|---|---|
| 前端 | React + Vite + TypeScript | `frontend/` 已建立 MVP 工作台 |
| 后端 | FastAPI | `backend/` 已建立 API、模型、服务和测试 |
| 数据库 | PostgreSQL | `docker-compose.yml` 已配置，开发默认可用 SQLite |
| 队列/缓存 | Redis | `docker-compose.yml` 已配置 |
| 部署 | Docker Compose + Caddy | `deploy/` 已配置反向代理和服务器初始化脚本 |
| Agent | Hermes 独立容器 | 仅做钉钉消息/自动化验证，不直接访问业务库 |
| 工程约束 | Ponytail | 仅用于开发和 review，不进入业务运行时 |

## 3. 后端模块

当前代码已经建立这些模块边界：

- `auth`：钉钉登录占位接口，本地先生成测试用户。
- `opportunities`：新品机会池，保留来源文件、sheet、行号、快照。
- `assignments`：分配预览、确认分配、改派。
- `claims`：销售认领、不认领校验。
- `reviews`：主管审核，第一阶段结束于通过、驳回、退回补充。
- `stocking`：后续备货决策占位模块，不作为第一阶段验收。
- `arrival`：后续到货记录占位模块，不作为第一阶段验收。
- `summary`：后续四周总结占位模块，不作为第一阶段验收。
- `notifications`：通知日志和 `dedupe_key` 去重。
- `admin`：人员映射、角色配置。

## 6. 第一版刻意不做

- 不做 Kubernetes。
- 不做复杂 BI。
- 不做全量在线表写回，先保留写回开关。
- 不做广告费、缺货、1-4 周精细监控的完整自动化。
- 不把 Ponytail 或 Hermes 当成业务系统依赖。
