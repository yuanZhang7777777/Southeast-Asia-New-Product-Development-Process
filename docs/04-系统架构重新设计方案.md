# 东南亚新品流程协同平台系统架构重新设计方案

> 日期：2026-06-29  
> 状态：MVP 技术方案已落地为项目骨架，业务细节继续按模块确认。

## 1. 系统定位

本系统是钉钉在线表、ERP、库存、销售数据之上的“新品流程协同层”，不替代 ERP，也不把第一版做成大而全 BI 平台。

第一阶段业务验收只跑通：

```text
入池 -> 分配 -> 销售认领 -> 主管审核
```

> ⚠️ 口径校正（2026-06-29 运营视角）：本平台是“两段主流程 + 一座外部桥”。**MVP = 前段，止于“出可备货清单”**；审核通过应产出**可备货清单（导出，对齐《海外仓备货申请表》）**，而不是在平台内自动生成内部备货草稿（现骨架在审核通过时自动按 `单销×30` 建 `stocking_request`，需改）。
>
> **备货 / 采购 / 供应链 / 下单 = 外部桥**，平台只导出清单 + 登记状态 + 等到货通知回流，不做成内部工作流。

`备货`、`到货`、`四周总结` 相关模块当前只是技术占位，不作为第一阶段验收范围；其中 `stocking` 应改造为“导出 / 外部状态登记”，`arrival` 是后段（Phase 2）的到货通知回流入口。完整端到端链路见 `docs/08-端到端流程链路规划.md`。

## 2. 技术栈

| 层级 | 选型 | 当前落地 |
|---|---|---|
| 前端 | React + Vite + TypeScript | `frontend/` 已建立 MVP 工作台 |
| 后端 | FastAPI | `backend/` 已建立 API、模型、服务和测试 |
| 数据库 | PostgreSQL | `docker-compose.yml` 已配置，开发默认可用 SQLite |
| 队列/缓存 | Redis | `docker-compose.yml` 已配置 |
| 部署 | Docker Compose + Caddy | `deploy/` 已配置反向代理和服务器初始化脚本 |
| Agent | Hermes 独立容器 | 仅做钉钉消息/自动化验证，不直接访问业务库 |
| 工程约束 | Ponytail | 仅用于开发和 review，不进入业务运行时 |

## 3. 后端模块

当前代码已经建立这些模块边界：

- `auth`：钉钉登录占位接口，本地先生成测试用户。
- `opportunities`：新品机会池，保留来源文件、sheet、行号、快照。
- `assignments`：分配预览、确认分配、改派。
- `claims`：销售认领、不认领校验。
- `reviews`：主管审核，第一阶段结束于通过、驳回、退回补充。
- `stocking`：**外部桥**——应改造为“导出可备货清单 + 登记外部状态”，不是平台内部备货决策工作流；不作为第一阶段验收。
- `arrival`：后段（Phase 2）到货通知回流入口占位，不作为第一阶段验收。
- `summary`：后段（Phase 4）四周总结占位，不作为第一阶段验收。
- `notifications`：通知日志和 `dedupe_key` 去重。
- `admin`：人员映射、角色配置。

以下表已在 SQLAlchemy ORM 中建模（**含后段 / 外部桥占位表，并非都是 MVP 必需**：`stocking_request` 为导出/桥、`supply_chain_quote` 为外部只读、`arrival_record` / `four_week_summary` 为后段）：

| 表 | 用途 |
|---|---|
| `users` | 用户基础信息 |
| `role_mapping` | 人员、角色、小组、站点映射 |

| `source_record_snapshot` | 源表行快照 |
| `market_research_item` | 竞品、售价、月销等多行明细 |
| `flow_instance` | 流程实例 |
| `flow_task` | 待办任务 |

| `review_record` | 主管审核记录 |

| `four_week_summary` | 四周总结 |
| `notification_log` | 通知日志和去重 |
| `audit_log` | 审计日志 |

## 6. 第一版刻意不做

- 不做 Kubernetes。
- 不做复杂 BI。
- 不做全量在线表写回，先保留写回开关。
- 不做广告费、缺货、1-4 周精细监控的完整自动化。
- 不把 Ponytail 或 Hermes 当成业务系统依赖。
