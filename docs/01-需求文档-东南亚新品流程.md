# 东南亚新品流程需求文档

> 版本：v0.4  
> 日期：2026-06-29  
> 状态：需求框架草案，后续按讨论持续更新  
> 目标：站在运营部门视角，把东南亚海外仓新品做成“两段主流程 + 一座外部桥”的可追踪、可提醒、可复盘流程系统。

> ⚠️ 口径校正（2026-06-29，运营部门视角）：本流程是 **“两段主流程 + 一座外部桥”**，不是端到端线性主流程。
> 平台只拥有 **前段（入池→分配→认领→审核→出可备货清单）** 与 **后段（到货承接→刊登→监控→四周总结）**；
> **备货 / 采购 / 供应链核价 / 下单 = 外部桥**，平台只导出清单 + 登记状态，**不建运营待办**。**MVP = 前段，止于“出可备货清单”。**
> 角色口径：**销售 = 运营、主管 = 运营负责人**；不认领原因 = 自定义文本；开品邮件 = 条件性状态标记。
> 下文 §5 主流程、§6.5–6.7 备货节点按本校正理解（备货节点为外部桥，非平台待办）。

## 1. 背景和问题

当前新品流程散在多个 Excel / 钉钉在线表 / SOP / 原型页面里。主要问题是：

- 流程节点多，容易漏处理。
- 销售、主管、开发/采购、运营之间责任边界需要明确。
- 哪些字段由系统带出、哪些字段由人填写还没有完全固定。
- 到货、缺货、广告费、1-4 周达成等异常需要及时提醒。
- 后续需求会持续变化，需要一份长期维护的项目需求文档，避免切换会话后丢失上下文。

## 2. 建设目标

第一阶段目标不是做重型 ERP，而是做一个围绕钉钉酷应用的轻流程系统：

1. 统一承接新品业务流程。
2. 每条新品机会都有状态、责任人、截止时间和处理记录。
3. 销售、主管、开发/采购、运营只看到自己该处理的任务。
4. 钉钉酷应用作为处理入口。
5. 钉钉待办用于“必须处理”的任务。
6. 钉钉工作通知用于“提醒、告警、汇总、升级”。
7. 所有提醒都要有去重和日志，避免重复打扰。
8. 字段、规则、责任人要能随业务讨论逐步调整。

## 3. 总体技术方案

推荐采用：

```text
钉钉在线表 / ERP / 广告平台 / 库存数据
        ↓
数据同步与规则服务
        ↓
流程实例 / 流程任务 / 字段值 / 提醒规则 / 通知日志
        ↓
钉钉酷应用页面 + 钉钉待办 + 钉钉工作通知
```

### 3.1 第一阶段方案

- 数据承载：先保留钉钉在线表 / Excel 作为业务数据来源。
- 流程状态：新增一套流程状态数据，不直接依赖单个在线表的颜色、筛选或人工备注。
- 后端服务：建议使用 Python FastAPI 或 Node.js 实现。
- 定时任务：定时同步在线表、ERP、库存、广告费等数据，并触发提醒规则。
- 前端入口：钉钉酷应用，提供“我的待办”“详情页”“提交表单”“主管审核”等页面。
- 通知方式：
  - 行动任务：创建钉钉待办。
  - 状态通知、异常告警、日报汇总：发送钉钉工作通知。

第一阶段业务验收范围修正为：

```text
入池 -> 分配 -> 销售认领 -> 主管审核
```

备货、采购、到货、刊登、1-4 周监控和四周总结属于后续阶段。完整端到端规划见 `docs/08-端到端流程链路规划.md`。

### 3.2 第二阶段方案

当流程和字段稳定后，再考虑：

- 把核心数据迁到 MySQL / PostgreSQL。
- 在线表只做协作视图、数据导出或报表展示。
- 增加权限模型、审计日志、规则配置页面。
- 增加 BI 看板：认领进度、备货进度、到货异常、精品成功率、缺货影响、广告投产等。

### 3.3 钉钉能力边界

当前设计依赖三类钉钉能力：

- 酷应用：作为员工在钉钉内处理流程的轻应用入口。
- 钉钉待办：用于需要明确责任人、截止时间、处理动作的任务。
- 工作通知：用于提醒、异常、汇总、升级。

实际开发前需要复核钉钉开放平台最新文档和权限申请项，尤其是待办、工作通知、用户身份、在线表读写、酷应用跳转参数。

参考官方入口：

- [钉钉开放平台](https://open.dingtalk.com/)
- [钉钉待办任务接口](https://open.dingtalk.com/document/development/add-dingtalk-to-do-task)
- [工作通知能力](https://open.dingtalk.com/document/development/work-notice-option)
- [文档酷应用接入](https://open.dingtalk.com/document/dingstart/access-document-coolapp)

## 4. 角色和责任

| 角色 | 主要责任 | 典型填写字段 | 典型系统带出字段 |
|---|---|---|---|
| 数据专员 / 系统 | 数据入池、数据同步、指标计算、异常识别 | 来源备注、人工修正项 | SKU、ItemID、国家、销量、库存、广告费、到货数量 |
| 主管 / 组长 | 分配任务、审核认领、确认异常升级、最终拍板 | 分配销售、审核状态、审核意见、驳回原因、主管确认 | 认领进度、逾期状态、异常汇总 |
| 销售 | 认领判断、销售调研、备货建议、四周总结 | 是否认领、认领单销、不认领原因、预计销量、销售反馈、总结结论 | 商品基础信息、成本、库存、历史销量、竞品基础数据 |
| 开发 / 采购 | 供应商、成本、采购、交期、备货异常处理 | 供应商、成本价、MOQ、交期、采购数量、异常原因 | 销售建议量、采购单状态、预计到货 |
| 仓库 / 库存系统 | 到货、入仓、上架、可售库存 | 异常备注 | 到货单号、到货日期、到货数量、可售库存 |
| 运营 | 刊登、广告、缺货处理、推品动作 | 刊登确认、广告处理动作、优化动作 | 店铺、Item、广告费、ROI/ACOS、库存天数 |

## 5. 主流程（两段 + 一桥）

源流程来自 `东南亚新品业务流程.xlsx`，但按运营部门视角应理解为“两段主流程 + 一座外部桥”，而非一条线性主流程：

```text
【前段·选品决策｜平台主流程·MVP】
  待认领数据(入池)
    → 分配 / 销售自领
    → 销售认领（认领填单销 / 不认领填理由）
    → 主管审核（通过 / 驳回 / 退回补充）
    → 出“可备货清单”

        │  导出可备货清单 +（条件性）开品邮件状态
        ▼
【外部桥｜非平台待办，只导出 + 登记状态】
  开发备货 → 销售备货 → 供应链核价 → 采购下单 → 开船·到港·到仓

        │  到货通知回流
        ▼
【后段·到货运营｜平台主流程·后续阶段】
  到货承接(到货通知，分集团内部 / 其他集团)
    → 刊登 → 库存/售价/链接/ItemID/店铺/站点确认
    → 二次调研（外老品热销 / 老品下跌 / 销售自主）
    → 推品（缺货监控 / 广告费监控；周度复盘循环 ≥4周）
    → 四周总结 → 后续动作
        ┄ 旁路：随时补货、滞销清仓
```

> 关键修正：**开发备货 / 销售备货 / 备货监控 / 采购不是平台主流程节点，而是外部桥**；平台只导出清单、登记状态、等到货通知回流。**到货刊登是后段重点。**

## 6. 流程节点需求

### 6.1 待认领数据

目标：把可认领新品、老品机会、销售自主调研机会统一进入待认领池。

建议模块：`新品待认领数据池`

关键字段：

- 数据ID
- 来源类型
- 来源链接/文件
- 入池日期
- 国家/站点
- SKU / ItemID
- 产品名称 / 图片
- 类目
- 热销 / 下跌指标
- 当前状态

待确认：

- 来源类型有哪些。
- SKU 和 ItemID 哪个作为主键。
- 去重规则如何做。
- 哪些数据自动入池，哪些由人工提交。

### 6.2 分配认领任务

目标：把待认领数据分配给具体销售或小组。

建议模块：`认领任务分配表`

关键字段：

- 任务ID
- 数据ID
- 分配日期
- 分配人
- 销售组 / 销售员
- 截止时间
- 任务状态

初步规则：

- 分配动作由主管 / 组长负责。
- 分配后立即给销售创建钉钉待办。
- 截止前提醒，逾期后升级主管。

待确认：

- 是否按主 SKU 整组分配。
- 是否按销售员当前任务量均衡分配。
- 是否考虑站点、类目、历史业绩、销售擅长品类。
- 是否允许一条机会分给多个销售。
- 改派规则和历史记录怎么保留。

### 6.3 销售认领

目标：销售判断是否承接，并补齐销售侧判断。

建议模块：`销售认领表`

关键字段：

- 认领结果
- 认领时间
- 目标站点 / 店铺
- 预计售价
- 预计销量
- 竞品链接 / 依据
- 备货建议量
- 不认领原因
- 补充说明

提醒规则：

- 分配后立即提醒销售。
- 销售未按时反馈，先提醒销售，再升级主管。
- 销售提交后，提醒主管审核。

待确认：

- 预计销量周期是日销、首周、四周还是首月。
- 备货建议量是否默认按认领单销乘以 30 天。
- 不认领原因是否必须从枚举选择。
- 是否要求上传截图或竞品链接。

### 6.4 主管审核

目标：确认销售认领是否合理，决定通过、驳回或退回补充。

建议模块：`认领审核表`

关键字段：

- 审核状态
- 审核人
- 审核时间
- 审核意见
- 驳回原因
- 补充截止时间

提醒规则：

- 销售提交后立即提醒主管。
- 审核超时提醒主管。
- 审核通过后触发开发 / 采购备货任务。
- 审核驳回后提醒销售补充。

待确认：

- 哪些场景必须主管审核。
- 审核通过后是否自动生成备货草稿。
- 驳回后是退回销售修改，还是关闭任务。

> 🔁 §6.5–6.7（开发备货 / 销售备货 / 备货监控）已校正为 **外部桥**：主要由开发 / 采购 / 供应链在平台外处理，平台第一版**不做成运营待办**，只做“导出可备货清单 + 按邮件模板导出字段 + 记录开品邮件是否已发 + 登记外部状态 + 到货通知回流”。下列字段保留作为**导出 / 登记**参考，不代表平台内部工作流。

### 6.5 开发备货（外部桥·仅导出/登记）

目标：确认供应、成本、MOQ、交期和采购执行（外部部门负责；平台只登记结果状态）。

建议模块：`开发备货表`

关键字段：

- 开发 / 采购负责人
- 供应商
- 成本价
- MOQ
- 交期
- 采购数量
- 采购单号 / 备货单号
- 预计到仓日

提醒规则：

- 审核通过后给开发 / 采购建待办。
- 预计到仓日前检查进度。
- 预计到仓日逾期未到货时提醒开发 / 采购和主管。

待确认：

- 开发备货和销售备货是否分两张表。
- 最终采购数量由谁确认。
- 成本价和采购单号从 ERP 自动带出还是人工维护。

### 6.6 销售备货

目标：销售确认首批需求和销售计划。

建议模块：`销售备货需求表`

关键字段：

- 首批需求量
- 目标售价
- 目标毛利 / 毛利率
- 预估售卖周期
- 备货确认状态

待确认：

- 销售是否直接填写备货申请。
- 练玉君 / 组长是汇总提交，还是可以修改备货数量、国家、仓库。
- 销售需求量是否需要主管审批。

### 6.7 备货监控

目标：监控采购、发货、预计到仓和异常。

建议模块：`备货监控表`

关键字段：

- 需求数量
- 已下单数量
- 已发货数量
- 未发货数量
- 预计到货日
- 备货状态
- 异常原因
- 处理人 / 处理动作

提醒规则：

- 正常进度不逐条提醒，进入日报 / 周报汇总。
- 备货逾期、数量不足、供应商异常时提醒责任人。
- 异常未处理超过阈值，升级主管。

### 6.8 到货监控

目标：监控本集团认领货品和其他集团相关货品的到货、上架、缺口。

建议模块：

- `到货监控表`
- `其他集团到货监控表`

关键字段：

- 到货单号
- 到货仓
- 到货日期
- 到货数量
- 上架数量 / 可售库存
- 认领归属
- 集团名称
- 其他集团到货数量

提醒规则：

- 到货后提醒销售 / 运营确认刊登和库存。
- 到货数量少于需求量时提醒开发 / 采购、销售和主管。
- 到货后超过 N 天未刊登，提醒运营并升级主管。

待确认：

- 到货触发点按入仓日期、上架日期还是可售日期。
- 其他集团到货监控是否共享同一数据源和权限。

### 6.9 二次调研

目标：基于外部老品热销、老品销量下跌、销售自主发现进行二次判断。

建议模块：

- `二次调研表`
- `外部老品热销监控表`
- `老品销量下跌监控表`
- `销售自主调研表`

关键字段：

- 外部商品链接
- 外部销量 / 热度
- 调研结论
- SKU / ItemID
- 近 7 / 14 / 30 天销量
- 销量跌幅
- 下跌原因
- 销售发现
- 建议动作

提醒规则：

- 外部热销 / 老品下跌先进入汇总池。
- 被主管分配后才创建销售或运营待办。
- 销售自主提交调研后，可提醒主管审核。

待确认：

- 热销和下跌阈值。
- 时间窗口取 7 天、14 天还是 30 天。
- 销售自主调研是否必须主管审核后才能进入具体调整。

### 6.10 具体调整

目标：把调研结论转成可追踪动作。

建议模块：`调整动作记录表`

关键字段：

- 调整类型
- 调整前值
- 调整后值
- 执行人
- 执行日期
- 预期影响
- 复盘日期
- 动作状态

提醒规则：

- 生成动作后给执行人创建待办。
- 到期未完成提醒执行人。
- 超过阈值未完成升级主管。

待确认：

- 哪些动作需要主管审批。
- 哪些动作销售 / 运营可以直接执行。
- 调整动作是否需要写回平台后台或在线表。

### 6.11 刊登运营

目标：监控刊登、库存、缺货、广告费和运营动作。

建议模块：

- `刊登运营表`
- `缺货监控表`
- `广告费监控表`

关键字段：

- 刊登链接 / ItemID
- 刊登日期
- 可售库存
- 库存天数
- 缺货天数
- 广告计划ID
- 广告花费
- 广告销售额 / 订单
- ROI / ACOS
- 广告异常处理动作

提醒规则：

- 到货后 N 天未刊登，提醒运营。
- 可售库存为 0 或库存天数低于阈值，提醒销售 / 运营。
- 广告费超阈值且 ROI / ACOS 异常，提醒运营。

待确认：

- 缺货定义。
- 广告异常阈值。
- ROI 和 ACOS 使用哪个作为主指标。

### 6.12 1-4 周监控实际达成情况

目标：看新品上线前四周目标和实际是否匹配。

建议模块：`新品1-4周达成监控表`

关键字段：

- 四周目标销量 / 销售额 / 毛利
- 第 1 周实际销量 / 销售额 / 毛利
- 第 2 周实际销量 / 销售额 / 毛利
- 第 3 周实际销量 / 销售额 / 毛利
- 第 4 周实际销量 / 销售额 / 毛利
- 广告费
- 达成率
- 异常标签
- 处理计划

提醒规则：

- 第 7 / 14 / 21 / 28 天生成周度检查。
- 低于目标阈值时触发异常处理待办。
- 正常达成进入汇总，不逐条频繁提醒。

待确认：

- 周起点按刊登日、首单日、上架日还是可售日。
- 达成主指标是销量、销售额、毛利还是组合判断。

### 6.13 四周总结

目标：复盘四周表现，判断缺货、广告、库存、售价等因素对结果的影响，并形成下一步动作。

建议模块：`四周总结表`

关键字段：

- 四周总销量 / 销售额 / 毛利
- 四周广告费 / ROI
- 是否达成
- 缺货影响判断
- 总结结论
- 后续动作
- 复盘人
- 主管确认

提醒规则：

- 第 28 天自动生成四周总结待办。
- 逾期未总结提醒复盘人。
- 继续逾期升级主管。
- 如果期间存在缺货天数，必须填写缺货影响判断。

待确认：

- 四周总结模板。
- 总结结论枚举。
- 是否自动生成下一轮补货 / 调整任务。

## 7. 提醒框架

### 7.1 提醒分类

| 类型 | 用途 | 推荐钉钉能力 | 示例 |
|---|---|---|---|
| 任务型提醒 | 必须有人处理 | 钉钉待办 | 销售认领、主管审核、备货确认、四周总结 |
| 异常型提醒 | 需要及时介入 | 工作通知 + 必要时待办 | 逾期、缺货、到货不足、广告异常、达成率低 |
| 状态型提醒 | 只需要知道 | 工作通知汇总 | 已到货、已下单、已刊登、审核通过 |
| 复盘型提醒 | 到时间才触发 | 钉钉待办 | 第 7 / 14 / 21 / 28 天检查和总结 |

### 7.2 提醒设计原则

1. 能汇总就不逐条打扰。
2. 必须处理的事项建待办。
3. 异常先提醒责任人，再按规则升级主管。
4. 每条提醒必须有去重 key。
5. 每次提醒都写入通知日志。
6. 同一任务在未发生状态变化前，不重复发送相同提醒。
7. 提醒文本必须带处理入口，点击进入对应酷应用页面。

### 7.3 初始提醒规则

| 触发点 | 接收人 | 方式 | 时机 | 升级 |
|---|---|---|---|---|
| 分配认领任务 | 销售 | 待办 | 分配后立即 | 逾期后主管 |
| 销售提交认领 | 主管 | 待办 | 提交后立即 | 审核超时提醒主管 |
| 主管驳回 | 销售 | 待办/通知 | 驳回后立即 | 补充超时主管 |
| 审核通过 | 开发/采购 | 待办 | 通过后立即 | 备货超时主管 |
| 预计到货延期 | 开发/采购、主管 | 工作通知 | 每日规则扫描 | 持续延期升级 |
| 到货完成 | 销售/运营 | 待办 | 到货后立即 | 未刊登升级主管 |
| 到货数量不足 | 开发/采购、销售、主管 | 工作通知 | 到货扫描时 | 主管跟进 |
| 缺货 | 销售/运营 | 工作通知/待办 | 库存低于阈值 | 持续缺货主管 |
| 广告异常 | 运营 | 工作通知 | 广告数据同步后 | 持续异常主管 |
| 第 7/14/21 天检查 | 销售/运营 | 待办 | 到期自动生成 | 逾期主管 |
| 第 28 天总结 | 销售/运营 | 待办 | 到期自动生成 | 逾期主管 |

## 9. 酷应用页面框架

第一阶段建议页面：

1. `我的待办`
   - 按当前钉钉用户展示需要处理的任务。
   - 支持按流程节点、逾期状态、优先级筛选。

2. `任务详情`
   - 展示商品信息、来源、历史数据、当前节点、字段表单、处理记录。
   - 所有钉钉待办点击后进入这里。

3. `分配台`
   - 给主管 / 组长使用。
   - 展示待分配数据、销售任务量、主 SKU 分组、分配建议。

4. `销售认领页`
   - 给销售填写是否认领、认领单销、预计销量、备货建议、不认领原因。

5. `主管审核页`
   - 给主管审核通过、驳回、退回补充。

6. `备货确认页`
   - 给开发 / 采购 / 销售确认供应、成本、采购数量、预计到仓。

7. `到货操作页`
   - 给销售 / 运营处理到货后的刊登、库存、调价。

8. `二次调研与调整页`
   - 展示热销、下跌、库存、广告等异常，填写调研结论和调整动作。

9. `1-4周监控页`
   - 展示目标、实际、广告、库存、异常标签和处理计划。

10. `四周总结页`
    - 填写总结结论、缺货影响判断、后续动作和主管确认。

## 10. 权限原则

初步权限：

- 销售：只能处理分配给自己的任务，查看自己相关记录。
- 主管 / 组长：查看和处理本组任务、审核、分配、异常升级。
- 开发 / 采购：查看通过审核后进入备货的任务，维护供应和采购信息。
- 运营：查看到货、刊登、缺货、广告、推品相关任务。
- 管理员 / 数据专员：维护数据同步、人员映射、规则配置、异常修正。

待确认：

- 是否允许跨组查看。
- 其他集团到货监控的数据权限。
- 主管是否可代销售处理。
